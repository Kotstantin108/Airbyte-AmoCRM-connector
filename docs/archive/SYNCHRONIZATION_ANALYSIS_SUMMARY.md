# РЕЗЮМЕ: Анализ синхронизации L2 → Production

## 📌 Краткий обзор анализа

Проведён глубокий анализ кода синхронизации данных между L2 (Analytics) и Production (amo_support_schema). Выявлены **12 проблем**, из которых **4 КРИТИЧЕСКИЕ**.

---

## 🔴 КРИТИЧЕСКИЕ ПРОБЛЕМЫ (Требуют срочного решения)

### 1. **Несоответствие между доменами** — ПОЛНОЕ ОТСУТСТВИЕ
- **Описание**: Скрипт `fdw_sync_multidomain.sql` использует таблицы `amo_support_schema.concepta_*` и `amo_support_schema.entrum_*`, но они **никогда не создаются** в bootstrap-скрипте
- **Последствие**: Синхронизация падает с ошибкой "relation does not exist"
- **Решение**: Выполнить скрипт `sql/CRITICAL_FIXES.sql`

### 2. **Недостаточная проверка при Ghost Busting**
```sql
IF (SELECT COUNT(*) FROM _meta_leads) = 0 AND v_prod_leads_count > 100 THEN
```
- **Проблема**: Если FDW вернёт 5 строк вместо 5000, проверка не сработает
- **Риск**: Удаление 4995 лидов БЕЗ ОТКАТА
- **Решение**: Изменить на проверку `% потерь > 20%` (см. CRITICAL_FIXES.sql)

### 3. **Отсутствие PRIMARY KEY на Production**
- **Проверка**: `SELECT * FROM amo_support_schema.concepta_leads;` (должны быть ключи)
- **Если нет ключей**: ON CONFLICT упадёт с ошибкой
- **Решение**: `ALTER TABLE ... ADD PRIMARY KEY (lead_id);`

### 4. **Clock Skew между L2 и Production**
```sql
SELECT MAX(started_at) - INTERVAL '15 minutes'  -- ❌ Рискованно
-- Должно быть:
SELECT MAX(started_at) - INTERVAL '1 hour'      -- ✅ Безопасно
```
- **Риск**: Пропуск данных с меткой _synced_at в "мертвой зоне"
- **Решение**: Изменить интервал с 15 мин на 1 час (UPSERT защитит от дублей)

## 🟡 СРЕДНИЕ ПРИОРИТЕТЫ

- Обработка очень больших батчей (>100K лидов за раз)
- Таймаут на долгие транзакции (statement_timeout)
- Pre-flight проверка доступности FDW

## ✅ ЧТО УЖЕ ХОРОШО РЕАЛИЗОВАНО

✓ Advisory locks (защита от параллельных запусков)  
✓ Dead Letter Queue для ошибок триггеров  
✓ Tombstone Shield для отслеживания удалений  
✓ Проверка на большие батчи (>5000 строк)  
✓ Корректная логика UPSERT с проверкой updated_at
